# GENESIS.md — System of Systems Operating Doctrine v0.1

## 1. Purpose

This system exists to prevent death spirals and promote virtuous spirals by organizing knowledge, judgment, capability, and action.

Core axiom:

> Infinite Depth. Finite Ontology. Stable Core. Adaptive Edge.

The system replaces infinite todos with structured objects:

* Affairs: what must be protected.
* Interests: what can be pursued.
* Missions: coordinated transformations of affairs, interests, and resources.
* Operations: execution containers.
* Routines: repeating cycles.
* Regimens: collections of routines.
* Reviews: structured learning from contact with reality.

## 2. Top-Level Architecture

| ------------ | ----------------------------------- | ---------------------------------------- | ------------------- |

## 3. The Four States

| State               | Question                   | Contains                                                                         |
| ------------------- | -------------------------- | -------------------------------------------------------------------------------- |
| State of the Art    | What is true / best known? | Sources of Volatility, Domains, Crafts, Models, Research, Signals                |
| State of Affairs    | What matters now?          | Affairs, Interests, Resources, Missions                                          |
| State of Functions  | What can the system do?    | War Gaming, Sensing, Planning, Preparation, Execution, Review                    |
| State of Operations | What is happening now?     | Horizontal Operations, Vertical Operations, Routines, Regimens, Agenda, Calendar |

Rule:

> Only these four get “State of” status. Everything else is a subview.

## 4. Architectural Principles

| Principle   | Definition                                                 |
| ----------- | ---------------------------------------------------------- |
| Hierarchy   | Objects are organized into levels.                         |
| Nestedness  | Every object belongs inside a parent context.              |
| Fractality  | Patterns repeat across scales.                             |
| Inheritance | Child objects inherit relevant parent properties.          |
| Recursion   | Operations can apply to their own outputs.                 |
| Meta        | The system can govern and improve itself.                  |
| Emergence   | Higher-order outcomes arise from lower-order interactions. |

Compression:

> Infinite depth comes from hierarchy, nestedness, fractality, inheritance, recursion, and meta-governance.

## 14. Mission

A mission is a coordinated transformation of affairs, interests, and resources through time, respecting path dependence, scale dependence, and interdependence, to move the system from its current state toward a preferred trajectory.

| Component         | Meaning                                      |
| ----------------- | -------------------------------------------- |
| Current State     | Where the system is now                      |
| Target Trajectory | Direction of improvement                     |
| Roadmap           | Path-dependent sequence                      |
| Blueprint         | Scale-dependent and interdependent structure |
| Affairs           | Downside exposures to protect                |
| Interests         | Upside options to pursue                     |
| Resources         | Assets and liabilities to allocate           |
| Operations        | Execution containers                         |
| Reviews           | Learning loop                                |

Mission is not a todo list. Mission is a trajectory controller.

Purpose:

* Prevent death spirals.
* Promote virtuous spirals.

## 15. War Gaming

War Gaming is the universal decision function.

It can be applied recursively:

| Input              | Output              |
| ------------------ | ------------------- |
| War Game Domain    | Domain map          |
| War Game Craft     | Doctrine update     |
| War Game Affair    | Hedge plan          |
| War Game Interest  | Edge plan           |
| War Game Mission   | Roadmap + blueprint |
| War Game Operation | Execution plan      |
| War Game Protocol  | Protocol update     |

Core questions:

| Question                      | Purpose                |
| ----------------------------- | ---------------------- |
| What is at stake?             | Identify exposure      |
| What is at risk?              | Identify downside      |
| What is the upside?           | Identify optionality   |
| What is the deadline?         | Identify time pressure |
| What signals matter?          | Identify evidence      |
| What resources are available? | Identify capability    |
| What protocol applies?        | Select action          |
| What is the abort condition?  | Define exit            |

## 16. War Gaming Gates

| Gate            | Question                                    | Prescription                                 |
| --------------- | ------------------------------------------- | -------------------------------------------- |
| Ergodicity Gate | Can the agent survive the sequence?         | If no, hedge or abort                        |
| Convexity Gate  | Is upside larger than downside?             | If yes, consider edge                        |
| Jensen Gate     | Does volatility help or hurt this exposure? | Long volatility if helpful; hedge if harmful |
| Kelly Gate      | What is the proper size of exposure?        | Size by survival, not excitement             |
| Abort Gate      | What condition forces exit?                 | Define before entry                          |

Technical placement:

* Ergodicity protects sequence survival.
* Jensen detects whether volatility helps or hurts.
* Convexity selects interests.
* Kelly sizes exposure.
* Abort prevents ruin.

## 25. Validation Tests

### Coherence Test

Every object must have one primary home.

Example:

Craft = Intelligence.
Code of Conduct = Directorial.
Protocol = Executive.

### Consilience Test

The same architecture must work across fitness, business, social life, learning, family, and statecraft.

### Emergence Test

The system should produce agency, status, capability, resilience, and optionality as emergent outcomes, not as shallow direct goals.

### Scalability Test

If a new need appears, prefer:

field → tag → relation → view → config

before adding a new entity type.

### Field Test

For any real situation, the system must answer:

1. Is this an Affair, Interest, Mission, Operation, Routine, or Signal?
2. What is at stake?
3. What is at risk?
4. What is the upside?
5. What gate applies?
6. Hedge, Edge, or Abort?
7. What protocol applies?
8. What is the next action?
9. What will be reviewed?
